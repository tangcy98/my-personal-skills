# 格式分析指南 & 周期性进度记录

## 第一部分：从过往报告学习格式

当用户提供过往报告时，skill 需要分析以下维度，以便对新内容应用相同的格式。

## 核心概念：一级和二级结构

在开始分析前，必须明确理解**一级结构**（主要话题）和**二级结构**（每个主要话题下的具体项目）：

### 一级结构示例
```
# 周报
├── UGC账号定义和策略        ← 一级
├── 优质内容定义和策略        ← 一级
├── 全站流量框架和围栏        ← 一级
└── 风险事项举手             ← 一级（不在其他一级下）
```

### 二级结构示例
```
## UGC账号定义和策略          ← 一级
  ├── 水位摸底               ← 二级
  ├── 模型训练               ← 二级
  ├── 扶持策略               ← 二级
  └── 活人UGC贡献度策略定义    ← 二级
```

## 1. 结构维度

### 主要章节
- 章节名称和顺序是什么？
  - 例：「本周进展」→ 「下周计划」 → 「存在问题」
  - 例：「Completed」→ 「In Progress」 → 「Blockers」
  - 例：「成果」→ 「进度」→ 「风险」

- 章节数量？
  - 通常 3-5 个主章节
  - 是否有子章节？

### 列表结构
- 使用符号？ ✓, ○, -, •, *
- 列表深度？ (一级、二级、三级)
- 缩进方式？

### 长度和粒度
- 每项多长？ (一句话 vs 一段话)
- 内容粒度？ (按任务 vs 按项目 vs 按时间)

## 2. 语言风格维度

### 正式程度
- 极度正式：「圆满完成了」「取得了突出成效」
- 专业正式：「完成了...」「修复了...」
- 相对随意：「搞定了...」「做完了...」

### 语气特征
- 第一人称 vs 被动语态
  - 「我完成了」vs 「已完成」vs 「完成了」
- 自信程度
  - 「必定」「确保」vs 「预计」「应该」vs 「可能」

### 细节程度
- 是否包含数字？ (「7 个 bug」vs 「若干 bug」)
- 是否包含原因？ (「因为...所以」)
- 是否包含影响？ (「导致...」「带来了...」)

### 措辞偏好
- 常用词汇和短语
  - 「已上线」vs 「已发布」vs 「已交付」
  - 「预计」vs 「计划」vs 「将」
  - 「待」vs 「需要」vs 「还未」

## 3. 格式维度

### 标题格式
- 中文 vs 英文
- 【】vs ** vs # vs 其他
- 中英文混用比例

### 项目符号
- ✓ ○ - • * □ ✕ ⚠ 等
- 是否混用多种符号？

### 日期/时间
- 是否标注周期？
- 日期格式？ (YYYY-MM-DD vs MM.DD vs 周一-周五)
- 是否标注百分比进度？

### 表格/数据
- 是否使用表格？
- 是否包含数据可视化？
- 指标类型？

## 4. 内容模式维度

### 「完成」项的典型表达
分析已完成项通常怎么写：

**例 1：** 「✓ 完成了支付模块重构，性能提升 40%」
- 用符号 ✓
- 「完成了」+ 主要工作 + 具体成果

**例 2：** 「○ 数据迁移进行中，已完成 70%」
- 用符号 ○
- 工作内容 + 百分比进度

**例 3：** 「完成了 5 个 bug 修复、2 个功能优化」
- 不用符号
- 量化所有内容

### 「进行中」项的典型表达
- 通常用什么符号？
- 百分比如何标注？
- 是否标注预计完成时间？

### 「计划」项的典型表达
- 是否只列标题，还是包含详情？
- 优先级如何标注？
- 时间如何表达？

## 5. 针对不同报告类型的分析

### 工程/技术报告
- 是否强调 bug 修复数量？
- 是否标注性能指标？
- 是否区分不同类型工作？(coding, review, ops)
- 是否包含技术债或技术风险？

### 产品/运营报告
- 是否强调数据指标？
- 是否包含用户反馈？
- 是否强调业务价值？
- 是否包含对标/竞品数据？

### 设计/运维/其他
- 独特的重点是什么？
- 特殊的术语或表述？

## 6. 一致性检查

- 同类项目的表述是否一致？
- 章节的「权重」是否平衡？(不要一个章节特别长)
- 整体的「节奏」如何？(是否有松有紧)

## 语义映射指南 — 理解内容应该放在哪里

用户的口语化表达往往不明确，需要通过语义理解映射到正确的一级和二级位置：

### 常见映射例子

**「模型部署完毕」**
- 属于：**模型训练** 二级项（不是新的一级或二级）
- 理由：部署是模型完成过程的一部分

**「新标签流量策略实验启动」**
- 属于：**扶持策略** 二级项（不是新的一级或二级）
- 理由：流量分配和实验是策略执行的一部分

**「发布意图模型标注启动」**
- 属于：**优质定义** 二级项下的内容（不是新的二级项）
- 理由：模型标注是定义过程的一部分

**「优质短文试水方案已明确」**
- 属于：**扶持策略** 二级项（可能包含多行信息）

## 应用指南

当需要基于过往报告格式化新内容时：

1. **先分析**过往报告的一级和二级结构
2. **理解语义**：哪些看似新的内容实际上属于现有的二级项
3. **提取关键特征**：「这个报告用 ✓ 符号、很正式、强调数字」
4. **分类新内容**到同样的一级和二级结构中
5. **用相同的语言和格式**改写新内容
6. **处理新项**：如果有不符合现有结构的内容，询问用户应该在哪里

### ⚠️ 关键防守原则

**绝对不要自动补充过往报告中有但用户本周没提到的内容：**

| 错误做法 | 正确做法 |
|---------|--------|
| 过往有「低差笔记打压」，用户没提，我就自动加上 | 只有用户明确提到「低差笔记打压」的更新，才加入 |
| 看到过往结构完整，就把过往的所有章节都用上 | 只包含用户本周实际提到或讨论的一级主题 |
| 用过往报告的内容来「补全」本周报告 | 本周报告就是空白的该部分就空白，不要填充 |

**判断标准：用户提到了吗？**
- 用户明确说过吗？ → 包含 ✓
- 用户隐含指代吗？ → 包含 ✓  
- 用户完全没提？ → 不包含 ✗（即使过往报告有）

这样，输出就会是「更像那个人的报告」而不是「套用通用模板」。

## 第二部分：周期性进度记录与聚合

### 为什么需要持久化进度记录？

传统周报的痛点：
- 需要在周报日集中回忆整周的工作
- 容易遗漏关键信息
- 没有时间戳，难以追踪进度
- 需要手动重复整理

解决方案：**全周累积，周报日聚合**

### 进度记录格式

存储在项目根目录下的 `memory/weekly-progress-YYYY-MM-DD.md`：

```markdown
# 周进度记录 (2026-03-16 to 2026-03-20)

- [Mon 3.17] 模型训练：完成了模型准召对齐，准确 90%、召回 85%
- [Mon 3.17] 扶持策略：新标签流量实验筹备中，最快本周内启动
- [Wed 3.19] 优质定义：产品内部方向共识，后续聚焦「发日常」推进
- [Wed 3.19] 扶持策略：外显笔记编辑权限设置入口已详评，CNY后合入
- [Fri 3.21] 被动促产：消费模型建模完成，本周内开实验
```

### 聚合逻辑

在周报日，skill 会：

1. **读取全周进度记录**
2. **按主题分组** — 将相同主题的更新归到一起
   ```
   模型训练：
   - 完成了模型准召对齐
   - 准确 90%、召回 85%
   ```

3. **去重** — 移除重复或冗余信息
4. **排序** — 按重要性或时间排序
5. **合并相关更新** — "推全"和"队列独立"这样的相关更新合并成一条
6. **映射到结构** — 放入过往报告的一级和二级位置

### 关键原则

- **不修改原始记录** — 原始进度记录保持不变，便于未来追溯
- **聚合是单向的** — 从原始记录生成报告，不反向修改
- **周报日后清空** — 生成报告后，`weekly-progress-YYYY-MM-DD.md` 可选择保留或归档
- **下周新文件** — 每周自动使用新的进度文件（`weekly-progress-YYYY-MM-DD.md`）
